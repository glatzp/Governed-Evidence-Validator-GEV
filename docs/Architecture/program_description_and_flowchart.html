Governed Evidence Validator

Phase 2: Document Q&A Module Architecture & Execution Specification

System Concept & Philosophy

The Governed Evidence Validator (GEV) is a document grounding and verification system. It provides a strict governance layer around the execution of large language models (LLMs) to ensure that every fact, quote, or claim returned to a user is objectively traceable to an authoritative source document (such as a union grievance procedure, contract, or policy document).

GEV operates on a core set of non-negotiable principles:

1. LLM Proposes, Program Verifies

The LLM is treated as an untrusted agent. It may propose facts or claim to find citations, but it has no authority to issue verdicts. Every proposed fact is cross-checked by the deterministic program layer using the Canonical Document Model.

2. Manual Hand-Off Scaffolding

The program never connects to an external LLM API at runtime. GEV structures task packets, and the user acts as the secure mediator, carrying prompts to the model of their choice and bringing response payloads back. This guarantees absolute data containment and model independence.

Canonical Document Model & Segmentation

To perform rigorous string and offset matching, GEV translates uploaded documents into an immutable CanonicalDocument representation containing structured EvidenceUnit components.

Text Segmenter

Sections are divided into paragraph-sized chunks (Evidence Units) with a maximum size of 2,500 characters. The GEV segmenter splits text with hierarchical priority:

  1. Splits on blank lines (double newlines) first to establish paragraph boundaries.
  2. If a paragraph exceeds 2,500 characters, it splits on line breaks.
  3. If line breaks are still oversized, it splits on sentence boundaries as a last resort.

Strict Index Traceability

Each Evidence Unit is mapped with absolute character indices (char_start and char_end) relative to the canonical concatenated document, a SHA-256 content hash, and its source file metadata. Verified claims are bound to a formal Provenance record. If an offset discrepancy is detected, the program raises a VerificationError and aborts verification.

Document Quality Diagnostics

To preserve evidentiary integrity, GEV automatically runs a Document Quality Diagnostics evaluation immediately after a document is loaded. This check runs silently in the background to ensure that the document's structure and formatting will not interfere with the validator's deterministic matching algorithms.

Silent Background Diagnostics & Audit Logging

The diagnostics analyze five core quality signals to flag formatting issues that could hinder text matching:

  • Broken Word Rate: Checks if words are frequently split across lines or extracted word-by-word (common in scanned or poorly structured PDFs).
  • Line Length: Flags average line lengths under 15 characters, indicating abnormal vertical alignment.
  • Empty Sections: Identifies parsed sections containing no extractable text content.
  • Malformed Tokens: Measures occurrences of replacement characters () or non-standard ASCII characters (OCR artifacts).
  • Document Size: Warns if the document text is very short (under 500 characters).

Governance Integration: The diagnostics do not block user action or show banners in the Web UI. Instead, the resulting DiagnosticReport assigns a quality rating (GOOD, DEGRADED, or POOR) and writes details directly to the append-only audit trail via `DOCUMENT_QUALITY_CHECKED`. If quality is POOR, GEV logs a `DOCUMENT_QUALITY_POOR` audit event, which is mapped directly to the PUBS Environmental Reconstruction failure pattern to serve as a silent, permanent record of potential input quality risks.

Interactive Verification Architecture

The system supports two interaction architectures: the consolidated Single-Trip Session Document (preferred for Web UI to combine LLM calls) and the Step-by-Step Interactive CLI (for detailed debugging).

The Three-Step Interactive Wizard (Web UI)

The user-facing experience is split into three distinct setup pages, followed by a unified Session & Validation dashboard. A progress panel on the left maps the user's path, updating as they transition through the system.

Step 1 — Source Material
The user uploads documents (.txt or .pdf files). The GEV supports loading multiple files, consolidating them in memory as a single authoritative corpus. Behind the scenes, GEV automatically evaluates document quality and compiles a DiagnosticReport to track input risks.
Step 2 — Topic Selection
GEV presents a list of dynamically parsed section headers or lets the user enter a manual topic. This defines the outline scope for the grounding verification.
Step 3 — Specific Question
The user inputs their specific question and initiates the verification session. GEV automatically compiles the prompts and document text into a single unified session document.
Session & Validation Dashboard
The main hub for model interaction and validation. The user downloads the consolidated Session Document, copies the primer prompt into their LLM, lets the model complete both passes, and pastes the raw delimited response back into GEV to run verification and view the audit log.

Verification Vocabulary & Verdicts

Verdicts are computed deterministically. The program verifies if claims match source texts and classifies them as:

Pass
Verified verbatim quote. The proposed passage matches the source text above the configured precision threshold. Tied to a specific Evidence Unit and offset range. (Displayed as Pass in raw logs, mapped to GROUNDED in user interfaces).
Human Review — Evidence Found
The claim is a paraphrase, procedural instruction, or unknown type, but GEV successfully located corresponding evidence within the document (surfaced with match score). (Mapped to HUMAN_REVIEW_REQUIRED in user interfaces).
Review Needed — Conflict Detected
Warning issued if contradictory evidence is identified for the same claim across different sections or documents. (Mapped to HUMAN_REVIEW_REQUIRED with conflict warning details in user interfaces).
Fail
The claim could not be verified in the source text. Occurs either because a citation is unverified (failed string match) or the LLM failed to propose a supporting passage. (Mapped to CITATION_UNVERIFIED or CITATION_REQUIRED in user interfaces).
Absent
No responsive information or matching sections exist within the document for the user's question (Absence Detection).
